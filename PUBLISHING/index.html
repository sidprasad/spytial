    This guide provides complete instructions for publishing sPyTial to PyPI, with both automated and manual options.

    ü§ñ Automated Publishing Setup

    Prerequisites

    1. GitHub Secrets Required:
    2. PYPI_API_TOKEN - Your PyPI API token (Get one here)

    3. TEST_PYPI_API_TOKEN - Your Test PyPI API token (Get one here)

    4. How to Add Secrets:

    5. Go to GitHub ‚Üí Repository ‚Üí Settings ‚Üí Secrets and Variables ‚Üí Actions
    6. Click "New repository secret"
    7. Add both tokens

    Workflows Available

    1. Continuous Integration (.github/workflows/ci.yml)

    Triggers: Push to main/develop, Pull Requests

    What it does: - ‚úÖ Tests on Python 3.8-3.12 - ‚úÖ Code quality checks (black, flake8) - ‚úÖ Full test suite - ‚úÖ Build validation - ‚úÖ Installation testing

    2. Automated Publishing (.github/workflows/publish.yml)

    Triggers: GitHub Releases, Manual dispatch

    What it does: - ‚úÖ All CI checks plus comprehensive validation - ‚úÖ Core functionality tests (from copilot instructions) - ‚úÖ Package building and validation - ‚úÖ Automatic publishing to PyPI or Test PyPI - ‚úÖ Release summary generation

    Publishing Process

    Option A: Release-Triggered Publishing

    1. Prepare release: 

      git add .
git commit -m "Prepare for release v0.1.0"
git tag v0.1.0
git push origin main --tags

    2. Create GitHub release:

    3. Go to GitHub ‚Üí Releases ‚Üí "Create a new release"
    4. Select your tag (v0.1.0)
    5. Add release notes
    6. Click "Publish release"
    7. Publishing automatically starts!

    Option B: Manual Workflow Trigger

    1. Go to GitHub Actions
    2. Select "Publish to PyPI" workflow
    3. Click "Run workflow"
    4. Choose options:
    5. ‚òê Publish to Test PyPI (for testing)
    6. ‚òë Publish to PyPI (for production)

    üõ† Manual Publishing

    Prerequisites

    pip install build twine

    Step-by-Step Process

    1. Validate everything is ready: 

      python scripts/quick_publish_check.py

    2. Clean previous builds: 

      rm -rf build/ dist/ *.egg-info/

    3. Build the package: 

      python -m build

    4. Test on Test PyPI first (recommended): 

      twine upload --repository testpypi dist/*

# Test install from Test PyPI
pip install --index-url https://test.pypi.org/simple/ spytial
python -c "import spytial; print('Test install successful!')"

    5. Publish to real PyPI: 

      twine upload dist/*

    üêõ Troubleshooting

    Common Issues

    1. Version files (=23.0.0, etc.):
    2. These are from broken pip install commands
    3. They're now in .gitignore

    4. Remove with: rm -f "=*.*.*"

    5. Test failures:

    6. Some advanced features may have failing tests
    7. Core functionality still works

    8. Use quick_publish_check.py for essential validation

    9. Authentication errors:

    10. Ensure API tokens are correctly set in GitHub secrets

    11. Tokens need "Entire account" scope for publishing

    12. Build failures:

    13. Check pyproject.toml and setup.py are consistent
    14. Ensure all required files are in MANIFEST.in

    Getting Help

    • Check GitHub Actions logs for detailed error messages
    • Test locally with python scripts/quick_publish_check.py
    • Validate distribution with twine check dist/*

    üéØ Version Management

    Releasing New Versions

    1. Update version numbers in:
    2. setup.py (line with version=)

    3. pyproject.toml (line with version =)

    4. Create and push tag: 

      git tag v0.1.1
git push origin main --tags

    5. Create GitHub release or publish manually

    Version Numbering

    • Use semantic versioning: MAJOR.MINOR.PATCH
    • 0.1.0 ‚Üí 0.1.1 (patch: bug fixes)
    • 0.1.1 ‚Üí 0.2.0 (minor: new features)
    • 0.2.0 ‚Üí 1.0.0 (major: breaking changes)
